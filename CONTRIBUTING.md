# Contributing to odf-kit

Thanks for your interest in contributing. This guide explains how to propose
changes, what conventions to follow, and what to expect during review.

## Discuss before coding

**Please open an issue to discuss any change before writing code.** This applies
to all contributions, including bug fixes, features, and refactors. The reason
is simple: odf-kit has design constraints that aren't always visible from the
code (ODF spec compliance, zero-runtime-dependency rule beyond `fflate` and
`marked`, public API stability), and a brief discussion saves contributors from
investing in work that conflicts with those constraints.

For very small fixes (typos in docs, obvious one-character bugs), a pull
request without prior discussion is acceptable — but expect the maintainer to
ask questions before merging if scope is unclear.

## Design principles

odf-kit follows a small set of standing rules that all contributions must
respect:

- **Spec compliance over convenience.** Every ODT, ODS, or other ODF file
  generated by odf-kit must pass the OASIS ODF validator without errors or
  warnings. The pre-commit pipeline enforces this for committed test fixtures.
  Workarounds that produce technically-valid-but-spec-questionable output are
  rejected.
- **Zero runtime dependencies beyond `fflate` and `marked`.** No new runtime
  dependencies will be accepted without significant discussion. Dev
  dependencies are added only when they provide value disproportionate to the
  installed footprint they bring.
- **Public API is additive-only.** Anything that touches the shape of an
  exported function, type, or module path requires explicit discussion before
  coding. Renames, signature changes, and removals are off-limits without
  prior agreement.
- **No silent magic.** Behaviour that the documentation doesn't describe
  shouldn't exist. If a feature needs to be discoverable, it goes in the
  docs; if it doesn't go in the docs, it probably shouldn't exist.

## Development setup

Requirements:

- Node.js 22 or later
- npm 10 or later
- Java (any modern JDK) for the ODF and HTML validators

Setup:

```bash
git clone https://github.com/GitHubNewbie0/odf-kit.git
cd odf-kit
npm install
```

## Pre-commit pipeline

Every commit should pass the full pipeline before being pushed:

```bash
npm run format:check
npm run format       # if format:check reports issues
npm run lint
npm run build
npm test
npm run validate-html
```

These commands run in seconds to a few minutes locally. The maintainer also
runs them before every commit. If any step fails on your machine, please
investigate and resolve before opening a pull request.

## Test conventions

- Tests live under `tests/` (for library code) and `docs/tests/` (for the
  unified tool page).
- Test files use either `*.test.ts` or `*_test.ts` — both patterns are
  recognised by Jest.
- Add tests for any new functionality. Add a regression test for any bug fix.
- Library tests run in the default `node` environment. DOM-touching code in
  the unified tool page is smoke-tested manually rather than via jsdom — see
  the project's testing posture in the related state docs.

## Code style

- Code is formatted by **Prettier**. Run `npm run format` before committing.
- Code is linted by **ESLint**. Run `npm run lint` before committing.
- TypeScript strict mode is on. New code is expected to be strict-clean
  without type-coercion shortcuts.
- Imports follow the project convention of `.js` extension on relative paths
  (TypeScript ESM-with-extension style).

## Pull request process

1. Open an issue to discuss your proposed change (see "Discuss before coding"
   above).
2. Fork the repository and create a feature branch.
3. Make your change, including tests and any necessary docs updates.
4. Run the full pre-commit pipeline locally and confirm it passes.
5. Open a pull request referencing the issue from step 1.
6. Be prepared to iterate based on review feedback.

## Reporting security issues

Please do not open public issues for security vulnerabilities. See
[`SECURITY.md`](./SECURITY.md) for the reporting process.

## Code of conduct

Be respectful and constructive. Bad-faith conduct, harassment, or any conduct
that makes the project unwelcoming will result in the contributor being asked
to stop and, if necessary, blocked from the project.

## Questions

If you're unsure about anything in this guide, open an issue with the label
`question`. The maintainer is responsive and would rather answer five
questions than discover a contribution went in a direction that won't be
merged.
